# Development Principles

> **AI 도구 사용 가능합니다. 단, 제출물에는 본인의 판단과 검증 결과가 반영되어야 합니다.**

## 1. AI 활용 철학

### AI는 도구이며, 개발자가 주인이다
- AI가 제안한 코드를 **무조건 수용하지 말 것**
- **왜** 이런 방식으로 구현했는지 스스로 설명할 수 있어야 함
- AI의 제안이 항상 최선은 아님 - 비판적 사고 필수

### 직접 작성 원칙
```
✅ 권장: AI의 제안을 참고하되, 직접 코드를 작성하고 이해한다
❌ 금지: AI가 생성한 코드를 복사-붙여넣기만 한다
```

## 2. 코딩 원칙

### 이해하지 못하면 쓰지 말 것
- 복사한 코드는 반드시 **라인별로 해석**할 것
- 모르는 문법이 있다면 **공식 문서나 책으로 학습**할 것
- "작동하니까 된다"는 태도 금지

### 직접 고민하고 결정하기
- 설계 결정(아키텍처, 패턴, 라이브러리 선택)은 **스스로 내린다**
- AI에게 "어떻게 해?"보다 **"이렇게 하는 게 맞을까?"**라고 질문한다
- trade-off를 스스로 분석하고 선택한다

## 3. 검증 원칙

### 모든 코드는 본인이 검증한다
- AI가 생성한 코드도 **직접 테스트**한다
- edge case를 스스로 찾아본다

### 동작 확인의 의무
```
1. 작성한 코드를 직접 실행해볼 것
2. 예상한 대로 동작하는지 확인할 것
3. 에러가 발생하면 디버깅하여 원인 파악할 것
```

## 4. 학습 원칙

### 모르는 것은 질문하고 배운다
- AI에게 질문을 던진다:
  - "왜 이 방식을 쓰나요?"
  - "다른 대안은 없나요?"
  - "이 방식의 단점은 무엇인가요?"

### 공식 문서 우선
- **공식 문서**를 우선 참고한다
- Kotlin: [kotlinlang.org](https://kotlinlang.org/docs/home.html)
- Spring Boot: [spring.io](https://docs.spring.io/spring-boot/docs/current/reference/html/)
- Next.js: [nextjs.org](https://nextjs.org/docs)

## 5. 제출 원칙

### 본인의 코드임을 증명한다
- 제출물의 **모든 코드**에 대해 설명할 수 있어야 함
- 왜 이런 설계를 했는지, 왜 이 라이브러리를 썼는지 답할 수 있어야 함
- AI의 도움을 받았다면 **어떤 부분을 받았고, 어떻게 수정했는지** 기록한다

---

**Remember**: AI는 뛰어난 도구이지만, 최종 책임은 개발자에게 있다. 🎯
